# MI325X (gfx942) + DeepSeek-V4 support in infera â€” design

Date: 2026-07-22
Branch: `yihou.dev.mi325`

## Goal

Bring MI325X (gfx942 / CDNA3) support for the DeepSeek-V4 (dsv4) family into the
infera library under a clear, enforced support contract. The current
`yihou.dev.mi325` branch integrated dsv4-Pro FP4 support, but the mechanism must
change: it currently enables **source patches** to sglang/atom for FP4 dequant,
which violates the "no third-party patching" rule. This redesign replaces that
with a support matrix that is native-only per engine and fails fast on
unsupported combinations.

## Support matrix (the contract)

On gfx942 (MI325X / CDNA3), per dsv4 variant Ă— quantization Ă— engine:

| variant | quant | vllm | sglang | atom |
|---------|-------|------|--------|------|
| Pro     | fp4   | âś… native (triton auto-dequant) | âťŚ fail-fast | âťŚ fail-fast |
| Pro     | fp8   | âťŚ fail-fast | âś… env + CLI knobs | âś… env knobs |
| Flash   | fp4   | âś… native (triton auto-dequant) | âťŚ fail-fast | âťŚ fail-fast |
| Flash   | fp8   | âťŚ fail-fast | âś… env + CLI + MTP | âś… env + MTP |

Rationale (the *why*, for docs):

- **fp4 â†’ vllm only.** gfx942/CDNA3 has no native FP4 MoE kernel. vLLM's
  `triton_unfused` MoE backend upcasts fp4â†’bf16 in-kernel (`tl.dot_scaled`) with
  no source patch. sglang and atom have no such native path â€” they only ran fp4
  via source patches in the reference experiments, which we now forbid (rule 3).
  So sglang/atom on an fp4 dsv4 checkpoint must fail fast, not silently degrade.
- **fp8 â†’ sglang/atom, NOT vllm.** vLLM was never validated for dsv4 fp8; sglang
  and atom run fp8 dsv4 natively (no patch).
- **Flash-fp8 needs MTP.** On gfx942 the dsv4-Flash compressed-MQA *decode*
  kernel is broken: prefill (first token) is correct, decode diverges to garbage
  (sglang) / stops after one token (atom). Running decode through a speculative
  (EAGLE / MTP) path avoids the broken plain kernel. Pro-fp8 is correct without
  it. Both checkpoints ship MTP weights (`num_nextn_predict_layers=1`).

## Scope of what infera injects (set-if-unset; operator always overrides)

Only functional, product-correct knobs. Experiment-specific memory workarounds
from the reference runs (`--cpu-offload-gb`, `--max-total-tokens`,
`--max-running-requests`, `--mem-fraction-static`, tiny `--context-length`) are
**not** injected â€” they were single-card VRAM band-aids, not defaults.

### fp8 sglang

- env: `HSA_NO_SCRATCH_RECLAIM=1`, `SGLANG_USE_ROCM700A=0`,
  `SGLANG_HACK_FLASHMLA_BACKEND=unified_kv_triton`, `AITER_BF16_FP8_MOE_BOUND=0`
- CLI (appended to forwarded `sglang_argv` if absent): `--attention-backend dsv4`,
  `--disable-shared-experts-fusion`
- Flash only, additionally: `--speculative-algorithm EAGLE
  --speculative-num-steps 3 --speculative-eagle-topk 1
  --speculative-num-draft-tokens 4`

Why each: `HSA_NO_SCRATCH_RECLAIM=1` is a gfx942-firmware requirement (dist-init
FATALs without it); `SGLANG_HACK_FLASHMLA_BACKEND=unified_kv_triton` because the
default MLA backend (tilelang) crashes its TVM compile on gfx942;
`SGLANG_USE_ROCM700A=0` and `AITER_BF16_FP8_MOE_BOUND=0` are the aiter fp8-MoE
settings the reference used; `--attention-backend dsv4` selects the sparse-MLA
path; `--disable-shared-experts-fusion` is the required dsv4 MoE customization.

### fp8 atom

- env: `HSA_NO_SCRATCH_RECLAIM=1`
- Flash only, additionally CLI (appended to `atom_argv` if absent):
  `--method mtp --num-speculative-tokens 3`

atom auto-detects fp8 quant from the HF config; the only firmware env it needs is
`HSA_NO_SCRATCH_RECLAIM=1`. MTP is atom's EAGLE-equivalent decode path.

### fp4 vllm

Nothing new. The existing native path already works (`VLLM_ROCM_USE_AITER=1` is
already defaulted by `apply_vllm_aiter_default`). No CLI injection.

## Components

### New module: `infera/engine/dsv4_gfx942.py`

Single home for all gfx942 dsv4 policy. This replaces the FP4-only
`apply_dsv4_gfx942_env_defaults` currently living in `rocm_rdma_env.py`; the RDMA
module returns to being RDMA-only. The logic (config detection + support-matrix
enforcement + env/CLI injection + fail-fast) has outgrown the RDMA module's
responsibility and is independently testable.

Public surface:

- `Dsv4Model` dataclass: `variant: "pro" | "flash"`, `quant: "fp4" | "fp8"`.
- `Dsv4UnsupportedError(RuntimeError)`: raised on an unsupported combo.
- `detect_dsv4(model_path) -> Dsv4Model | None`: reads `config.json` once from a
  local dir (never downloads). `None` if not a local dir, not dsv4
  (`model_type` not `deepseek_v4*` and no `index_topk`), or on any read error.
  - variant: `hidden_size >= 7168` or `num_hidden_layers >= 61` â†’ `pro`;
    `hidden_size == 4096` / `num_hidden_layers == 43` â†’ `flash`. Keyed off
    dimensions, never the directory name.
  - quant: `fp4`/`mxfp4`/`e2m1` in `quantization_config` â†’ `fp4`; `fp8`/`e4m3`
    â†’ `fp8`.
- `apply_gfx942_dsv4(model_path, *, engine, argv) -> list[str]`: the single call
  each engine `__main__` makes. No-op (returns `argv` unchanged) if not gfx942
  (reuses `rocm_rdma_env.is_gfx942`) or not a dsv4 model. Otherwise:
  1. enforce the support matrix â†’ raise `Dsv4UnsupportedError` on an unsupported
     `(engine, variant, quant)` with a message naming the working engine;
  2. apply env defaults set-if-unset (per matrix above);
  3. return `argv` with functional CLI flags appended set-if-unset (only for
     engines that take CLI injection: sglang, atom).

`is_gfx942` stays in `rocm_rdma_env.py` (arch probe is shared infra); the new
module imports it.

### Engine `__main__` wiring

- **sglang** `infera/engine/sglang/__main__.py`: replace the
  `apply_dsv4_gfx942_env_defaults(...)` call with
  `args.sglang_argv = apply_gfx942_dsv4(args.server_args.model_path,
  engine="sglang", argv=args.sglang_argv)`. Placed at the same point (after RDMA
  defaults, before `engine.start()`), so injected CLI reaches the launch_server
  subprocess and injected env is inherited.
- **atom** `infera/engine/atom/__main__.py`: replace the call with
  `args.atom_argv = apply_gfx942_dsv4(args.model, engine="atom",
  argv=args.atom_argv)` before `AtomEngine(...)`.
- **vllm** `infera/engine/vllm/__main__.py`: add
  `apply_gfx942_dsv4(args.model, engine="vllm", argv=[])` (env/enforce only; no
  CLI injection) so vllm fails fast on fp8 dsv4. Placed after the existing aiter
  default.

### `rocm_rdma_env.py` cleanup

Remove `apply_dsv4_gfx942_env_defaults` and `_is_dsv4_fp4_model` (moved/replaced
by the new module). Keep `is_gfx942`, `_compute_capability`, and all RDMA
functions. Update the three `__main__` imports accordingly.

## Five-way consistency (rule 1)

Every change lands as a set: doc (support matrix) â†” code (`dsv4_gfx942.py`
matrix) â†” image (Dockerfiles) â†” patch (fp4 patches removed) â†” test
(`test_dsv4_gfx942.py` asserts the matrix). The matrix is the single contract
all five express.
